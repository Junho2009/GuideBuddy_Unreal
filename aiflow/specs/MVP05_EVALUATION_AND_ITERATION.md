# MVP05：指导与练习效果评估迭代

## 本阶段目标

记录 LLM 给出建议或练习目标后的下一次或多次尝试，判断玩家的目标错误是否减少，并把结果用于迭代导玩策略和后续训练场配置。

本阶段完成后，项目应能回答：

**这条指导或练习有没有真的帮到玩家。**

## 本次一定要有

- 能把一次指导与后续一次尝试关联起来。
- 能把 `practice_objective` 或 Drill Spec 与后续尝试关联起来。
- 能生成 `evaluation.json`。
- 至少评估一个目标问题是否改善。
- 能记录改善、未改善或证据不足。
- 能为下一轮导玩提供简短反馈，例如继续练同一问题、降低提示强度或换一个目标问题。

## 优先评估指标

MVP 优先使用这些指标：

- 同一攻击类型的受击率是否下降。
- 面对同一前摇的响应时间是否更接近目标窗口。
- 是否继续死于同一原因。
- 存活时间是否提升。
- 错误治疗或贪刀次数是否下降。
- 玩家是否实际执行了建议中的动作。
- 针对性练习中是否达成 Drill Spec 的 `success_metrics`。

## 本次先不做

- 不做长期玩家画像。
- 不做多人排行榜。
- 不做复杂 A/B 测试平台。
- 不做云端数据同步。
- 不做跨关卡泛化评估。
- 不在本阶段生成训练场，只评估练习目标或指导是否有效。

## 验收方式

最低验收场景：

1. 玩家完成一次有诊断和指导的尝试。
2. 玩家再完成一次尝试。
3. 系统生成第二次尝试的遥测与诊断。
4. 系统把两次尝试与上一次指导关联起来。
5. 系统输出 `evaluation.json`，说明目标问题是否改善。

针对 Drill Spec 的扩展验收：

1. 系统读取一次带 `practice_objective` 的 coaching 输出。
2. 玩家完成一次普通尝试或练习尝试。
3. 系统把后续尝试与该练习目标关联起来。
4. 系统输出目标指标是否改善，以及下一轮应继续、降难度、升难度或换目标。

人工验收关注：

- 评估结论是否可理解。
- 改善或未改善是否有证据。
- 没有足够证据时是否明确说明。

自动验收关注：

- 能找到上一条指导。
- 能找到后续尝试。
- `evaluation.json` 可解析。
- 至少一个预设样例能输出 `improved`、`not_improved` 或 `insufficient_evidence` 中的一种。

## 依赖与衔接

- 依赖 MVP01 的多次尝试记录。
- 依赖 MVP02 的稳定诊断标签。
- 依赖 MVP03 的结构化指导输出。
- 本阶段结果可反哺后续 Prompt、诊断阈值、指导强度策略和 Drill Spec 难度参数。
- 后续迭代将依赖本阶段定义的目标指标和评价结果，决定训练场是否继续、升难度、降难度或更换目标。

## 备注

黑客松 demo 中，MVP05 是证明项目价值的关键：它能展示 AI 不只是“会说”，而是能围绕玩家的具体错误形成一个可衡量的学习闭环。针对性练习场上线后，MVP05 也是判断练习场是否真的有效的依据。
