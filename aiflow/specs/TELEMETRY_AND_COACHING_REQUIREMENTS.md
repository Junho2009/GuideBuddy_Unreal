# 战斗遥测、诊断与导玩要求

## 文档目的

本文规定 GuideBuddy MVP 四个阶段共同遵守的数据与导玩要求。

目标不是采集一切，而是让系统能回答一个核心问题：

**玩家这一次为什么失败，下一次最应该改哪一个动作。**

## 数据采集原则

- 采集语义事件，不采集无筛选的每帧位置流。
- 游戏侧记录事实，诊断层负责归因，LLM 负责表达。
- 事件应带时间戳，并能按一次玩家尝试聚合。
- 每条事件都应尽量包含主体、目标、来源资产或动作名、关键数值和战斗阶段上下文。
- 输出文件应可被脚本、LLM 前处理和人工复盘共同读取。

## 最低事件范围

MVP01 起，至少应逐步覆盖以下事件类型：

- 尝试开始：`attempt_start`
- 尝试结束：`attempt_end`
- 玩家死亡：`player_death`
- 敌人 ability / state / attack 激活
- 玩家关键输入：闪避、攻击、治疗、防御、锁定或其它当前战斗确有的动作
- 伤害事件：攻击来源、命中目标、伤害量、剩余生命、是否致命
- 属性变化：生命、耐力或体力类资源，以及其它影响能否行动的关键资源
- Trace 命中或等价命中事件
- 敌人阶段变化或明显战斗状态切换

## 推荐输出

推荐输出根目录：

```text
Saved/GuideBuddy/Telemetry/<run-id>/
```

最低文件：

- `combat_events.jsonl`：语义事件流，一行一个 JSON 事件。
- `attempt_summary.json`：本次尝试摘要，供诊断层和 LLM 前处理读取。
- `diagnosis.json`：诊断结果，从 MVP02 起生成。
- `coaching.json`：导玩建议，从 MVP03 起生成。
- `evaluation.json`：指导效果评估，从 MVP04 起生成。

## 诊断优先级

MVP 只优先解决少数能形成明显教学价值的问题：

- 闪避过早
- 闪避过晚
- 贪刀 / 过度承诺
- 资源耗尽导致无法应对
- 错误治疗时机
- 连续多次死于同一攻击或同一窗口

其它问题可以记录为 `unknown` 或 `insufficient_evidence`，不要为了显得聪明而强行归因。

## LLM 输入要求

LLM 不应直接消费完整原始事件流。

推荐输入包含：

- 玩家水平估计
- 本次尝试摘要
- 主要诊断标签
- 少量关键证据事件
- 目标问题的指标
- 本次只希望玩家改善的一个动作

## 导玩输出要求

每次导玩建议应保持短、具体、可尝试。

最低包含：

- 一句话诊断
- 一条证据
- 下一局只练一个动作
- 指导强度或玩家水平
- 诊断信心

MVP 默认采用死亡后卡片或尝试后复盘，不在战斗中频繁打断玩家。

## 评估要求

每条指导都应能在后续尝试中被评估。

优先评估：

- 同一攻击类型的受击率是否下降
- 面对同一前摇的响应时间是否更接近目标窗口
- 是否继续死于同一原因
- 存活时间是否提升
- 错误治疗或贪刀次数是否下降
- 玩家是否实际执行了建议中的动作

MVP 的效果判断以“目标错误是否减少”为主，而不是以 LLM 文案是否好听为主。
