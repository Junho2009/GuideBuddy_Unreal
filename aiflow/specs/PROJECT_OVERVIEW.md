# GuideBuddy_Unreal MVP 项目概览

## 项目一句话

GuideBuddy 是一个面向魂 like 战斗学习的 AI 导玩 MVP：它观察玩家在 `SampleDemoShowcaseMap` 中的实际战斗表现，判断玩家为什么受击或死亡，给出适合当前水平的一条具体改进建议，并把建议转成玩家可以进入的针对性单点练习场。

## 当前 MVP 目标

当前 MVP 不是完整游戏，也不是完整战斗系统重写。

本轮目标是跑通一条可信的学习闭环：

```text
玩家进行一次战斗尝试
  -> 系统记录语义化战斗事件
  -> 系统生成诊断信号
  -> LLM 生成死亡后或尝试后的导玩建议与最小练习目标
  -> 玩家再次尝试，或切换进入针对性练习场
  -> 系统评估目标错误是否减少
```

## 当前默认场景

- Unreal 项目：`TCF_Sample.uproject`
- 引擎版本：Unreal Engine 5.7
- 默认地图：`/Game/TCF_SampleDemo/SampleDemoShowcaseMap.SampleDemoShowcaseMap`
- 战斗框架：`TempestCombatFramework`
- 当前演示目标：玩家在示例战斗中至少完成一次可复盘的死亡或失败尝试。

## 核心体验目标

MVP 应让玩家感到：

- AI 知道我刚才具体怎么打的。
- AI 不是泛泛讲攻略，而是指出我这次失败的关键原因。
- AI 给的建议够短，下一局能马上尝试。
- AI 不只是告诉我该练什么，还能让我进入一个只练这个问题的训练场。
- AI 能看出我下一局有没有改善。

## 产品原则

- 先做死亡复盘卡，不先做实时教练。
- 先做少数高频错误，不先做全量招式百科。
- 先做语义事件和诊断摘要，不把原始每帧数据直接丢给 LLM。
- 先让 LLM 输出结构化练习目标和 Drill Spec，不让 LLM 直接自由生成或修改 Unreal 场景。
- 先用现有 Blueprint 战斗资产跑通闭环，不把完整 Blueprint-to-code 作为第一阶段目标。
- 新增可分析逻辑、遥测、诊断、数据导出和导玩编排优先落在 Puerts + TypeScript 与结构化文本层；C++ 仅作为必要桥接或 native fallback。

## 当前不做

- 不做完整魂 like 训练关卡。
- 不在 MVP01-MVP04 中做真实可进入的练习场运行时；MVP04 只生成受控 drill spec / session，练习场切换与运行进入 MVP05。
- 不在练习场可进入、可评估之前接真实 LLM provider；真实 provider 放到后置阶段替换本地规则模板。
- 不做实时语音陪练。
- 不做战斗中频繁弹窗干预。
- 不做完整蓝图逻辑迁移。
- 不做多 Boss、多关卡、多角色适配。
- 不做长期玩家画像系统。
- 不做在线服务化后台。

这些都可以作为后续扩展，但不进入当前黑客松 MVP 的最低闭环。

## MVP 最低成功定义

当以下条件都成立时，可以认为 GuideBuddy MVP 竖切跑通：

1. 玩家在 `SampleDemoShowcaseMap` 中进行一次战斗尝试。
2. 系统生成可读的 `combat_events.jsonl` 与 `attempt_summary.json`。
3. 系统能把一次死亡或明显失败归因到至少一种可指导问题。
4. LLM 基于诊断摘要生成一条短建议和一个机器可读的最小练习目标，而不是凭空泛讲。
5. 系统能基于 Drill Spec 让玩家进入一个受控的针对性练习场。
6. 下一次尝试或练习后，系统能比较目标问题是否改善。

## 针对性练习场扩展目标

在基础闭环跑通后，GuideBuddy 应逐步支持：

```text
诊断信号
  -> LLM 选择最小练习目标
  -> 生成结构化 Drill Spec
  -> UE 侧按白名单模板创建并切换进入训练场
  -> 玩家完成练习
  -> 系统评估目标问题是否改善
```

此扩展的核心原则是：LLM 负责教学设计和参数建议，UE 侧负责稳定、受控、可验证的场景生成。Drill Spec 必须是机器可读的结构化配置，而不是自由文本关卡描述。
