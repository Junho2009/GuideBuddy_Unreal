# MVP03：LLM 导玩与练习目标闭环

## 本阶段目标

基于 MVP02 的诊断结果，让 LLM 生成适合玩家水平、低打扰、可执行的导玩建议，并输出机器可读的最小练习目标与 Drill Spec 草案。

本阶段完成后，项目应能回答：

**玩家下一局最应该怎么改，以及最小应该练什么。**

## 本次一定要有

- 能读取 `attempt_summary.json` 与 `diagnosis.json`。
- 能构造高信号 LLM 输入摘要，而不是把完整事件日志直接丢给 LLM。
- 能生成 `coaching.json`。
- 能生成或嵌入 `practice_objective`。
- 能生成受控的 `drill_spec_candidate`，供 MVP04 进一步验证和执行。
- 至少支持死亡后或尝试后的单张复盘卡内容。
- 每次建议只聚焦一个主要问题。
- 建议内容应包含：
  - 一句话诊断。
  - 一条证据。
  - 下一局只练一个动作。
  - 适合玩家水平的表达方式。
  - 诊断信心或提示强度。
  - 可验证成功条件。
  - Drill Spec 草案，且只使用白名单模板和参数名。

## 玩家水平与表达

MVP 至少区分三种表达强度：

- 新手：少术语，只讲可观察动作。
- 进阶：可以提反应时间、窗口和趋势。
- 熟练：可以提风险收益、后摇、贪刀和优化空间。

如果暂时无法真实判断玩家水平，可以先使用默认水平，并在输出里保留 `player_level_assumption`。

## 本次先不做

- 不做实时语音教练。
- 不做战斗中频繁弹窗。
- 不做多轮自然语言聊天系统。
- 不做复杂情绪识别。
- 不做攻略百科。
- 不让 LLM 直接决定游戏内奖惩或状态修改。
- 不让 LLM 直接自由创建 Unreal 场景、Blueprint、资产或控制台命令。
- 不在本阶段实际生成训练场；本阶段只生成结构化练习配置草案。

## 验收方式

最低验收场景：

1. 使用 MVP02 生成的诊断结果。
2. 调用 LLM 或等价的可替换生成器。
3. 生成 `coaching.json`。
4. 输出内容可转成死亡后复盘卡。
5. 输出内容包含机器可读 `practice_objective`。
6. 如果启用 Drill Spec 草案，草案必须只引用白名单模板和参数。

人工验收关注：

- 建议是否短。
- 建议是否具体。
- 建议是否与证据一致。
- 建议是否适合目标玩家水平。
- 建议是否不会同时要求玩家改太多事情。
- 练习目标是否足够小，玩家下一局能马上尝试。
- Drill Spec 草案是否符合诊断，而不是为了“好玩”偏离目标问题。

自动验收关注：

- LLM 输入摘要存在。
- `coaching.json` 可解析。
- 必需字段齐全。
- 输出没有脱离诊断结果编造不存在的攻击或事件。
- `practice_objective` 可解析且能映射到一个可评估指标。
- Drill Spec 草案不包含白名单之外的模板、敌人、参数或自由指令。

## 依赖与衔接

- 依赖 MVP02 输出可靠诊断。
- MVP05 将记录本阶段给出的建议和练习目标，并评估后续尝试中目标问题是否改善。
- MVP04 将读取本阶段输出的 Drill Spec 草案，并通过 UE 侧受控模板生成训练场。
- 如果外部 LLM 暂不可用，本阶段可以先用 mock provider 或本地规则模板占位，但文件结构和调用边界应保持一致。
