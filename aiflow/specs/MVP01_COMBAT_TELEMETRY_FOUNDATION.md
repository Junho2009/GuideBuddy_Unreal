# MVP01：战斗遥测基础

## 本阶段目标

在 `SampleDemoShowcaseMap` 中建立最小战斗遥测能力，让系统能记录一次玩家尝试中的关键事实，并产出可供后续诊断和 LLM 前处理读取的结构化文件。

本阶段完成后，项目应能回答：

**玩家这次尝试中发生了什么。**

## 本次一定要有

- 能识别一次玩家尝试的开始与结束。
- 能记录玩家死亡事件。
- 能记录玩家关键输入，至少包括当前战斗中可用的闪避与攻击；若治疗或防御存在，也应记录。
- 能记录敌人关键 ability / state / attack 激活事件，至少保留可读名称或资产路径。
- 能记录伤害事件，包括来源、目标、伤害量、剩余生命和是否致命。
- 能输出 `combat_events.jsonl`。
- 能输出 `attempt_summary.json`。
- 输出内容足够让人复盘死亡前数秒发生了什么。

## 本次先不做

- 不要求精确识别所有攻击前摇、命中窗口和恢复窗口。
- 不要求完整蓝图转代码。
- 不要求实时 UI 展示。
- 不要求 LLM 生成建议。
- 不要求判断玩家水平。
- 不要求跨多局聚合趋势。

## 验收方式

最低验收场景：

1. 启动 `SampleDemoShowcaseMap`。
2. 玩家进入战斗并产生一次可结束的尝试，最好包含死亡。
3. 系统在 `Saved/GuideBuddy/Telemetry/<run-id>/` 下生成事件与摘要文件。
4. `combat_events.jsonl` 中能看到按时间排序的关键事件。
5. `attempt_summary.json` 中能看到本次尝试开始时间、结束原因、死亡信息、最后若干事件和伤害摘要。

人工验收关注：

- 人读这些文件时，能大致还原死亡前发生了什么。
- 文件不是空壳，也不是只包含不可读对象地址。

自动验收关注：

- 文件存在。
- JSON / JSONL 格式可解析。
- 至少包含尝试开始、一次敌人动作或状态、一次玩家输入、一次伤害或死亡事件。
- run-id 不覆盖上一轮输出。

## 依赖与衔接

- 依赖现有 TempestCombatFramework 示例战斗可运行。
- 后续 MVP02 将读取本阶段输出的事件和摘要。
- 本阶段不要求修改战斗平衡或原有 Blueprint 行为。
